# 通联随记

## 1. 背景

在户外或移动场景中，业余无线电用户不方便手写通联记录。项目希望提供一个移动端通联记录助手，让用户在通联过程中通过语音实时记录交流内容，并由 AI 辅助整理为结构化 QSO 记录，后续可导出日志文件。

项目参考：

- <https://github.com/richcannings/qso-scribe>

参考项目当前更偏 CLI/核心处理流程。本项目目标是将类似能力产品化为移动 App。首版先支持 Android，后续支持 iOS，并在更长期版本中考虑鸿蒙兼容适配。

## 2. 产品定位

通联随记是一个移动端实时 QSO 记录助手。

第一版重点解决现场通联时的记录痛点：用户手动开始和结束一条 QSO。App 支持实时流式转写，也支持在供应商不具备实时流能力时先录音、结束后统一识别和结构化。用户确认字段后保存正式 QSO 记录，并可导出 ADIF、CSV 以及原始转写文本。

## 3. MVP 主流程

### 3.1 实时记录

1. 用户进入实时记录页面。
2. 首次使用实时记录前，用户必须选择转写处理模式：
   - 实时流式转写：供应商支持实时流接口时使用，边通联边生成转写和结构化草稿。
   - 结束后统一识别：通联过程中仅录音，用户结束 QSO 后再统一提交转写和结构化。
3. 首次使用实时记录前，用户必须选择 AI/转写失败处理策略：
   - 及时报错：转写或结构化失败时明确显示错误状态。
   - 静默降级：结构化失败时保留原始转写文本，不强打断用户。
4. 用户后续可以在设置中修改转写处理模式和失败处理策略。
5. 用户点击“开始通联”。
6. App 开始采集语音。
7. 如果当前模式为实时流式转写，App 在通联过程中进行实时转写并结构化更新当前 QSO 草稿。
8. 如果当前模式为结束后统一识别，App 在通联过程中仅展示录音状态、计时和草稿占位。
9. 用户点击“结束通联”。
10. 如果当前模式为结束后统一识别，App 在结束后提交音频进行转写和结构化。
11. 用户检查并补全字段。
12. 满足保存规则后，用户保存为正式 QSO。

### 3.2 QSO 边界

第一版由用户显式控制 QSO 边界：

- 用户手动开始一条 QSO。
- 用户手动结束一条 QSO。
- App 不自动判断一条 QSO 的开始或结束。

### 3.3 结构化记录

第一版目标是在实时流式转写模式下实时结构化，而不是只展示原始转写文本。结束后统一识别模式下，结构化发生在用户结束 QSO 后。

实时流式转写模式下，App 应在通联过程中持续维护当前 QSO 草稿。结束后统一识别模式下，App 应在识别完成后生成 QSO 草稿。两种模式至少覆盖正式保存所需的基础字段：

- 呼号
- 日期时间
- 频率或波段
- 模式
- 信号报告，至少包括发送和接收 RST

以下扩展字段进入 MVP，但不作为正式保存的强制字段；AI 未识别到时允许用户补充填写：

- 对方姓名
- QTH
- 备注
- 设备
- 天线

### 3.4 无线电话语音处理

第一版需要针对业余无线电话通联语境做基础优化，而不是把语音内容当作普通会议转写处理。

MVP 需要考虑：

- 呼号识别应支持常见 ITU / DX 拼读字母。
- 结构化时应尽量忽略明显无关闲聊，避免把寒暄、背景说明或环境噪声误写入关键字段。
- 对低置信度字段保留“需确认”状态，提醒用户保存前复核。

处理规则：

- 低置信度字段使用颜色提示，直观标记字段需要用户复核。
- 必须保留呼号识别相关的原始转写片段，便于用户回看判断；用户后续可以选择删除原始内容。
- 口头纠错不作为独立语音命令进入 MVP，优先通过结构化 Prompt 处理，让模型在最终 JSON 中修正口头更正。

### 3.5 保存规则

保存正式 QSO 时，以下字段必须存在：

- 呼号
- 日期时间
- 频率或波段
- 模式
- 信号报告，至少包括发送和接收 RST

如果字段不完整，保存前要求用户补全。

AI 生成的字段不能绕过用户确认。第一版不做“自动生成最终日志且无需用户确认”的流程。

### 3.6 实时页面

实时页面需要同时支持当前 QSO 处理和本次会话回看。

MVP 至少包括：

- 当前 QSO 草稿。
- 本次会话 QSO 列表。
- 字段点按编辑，至少能修改呼号、日期时间、频率或波段、模式、备注。
- 原始转写文本查看入口。

### 3.7 失败处理

第一版不固定单一失败处理方式，而是在首次使用实时记录前让用户选择策略。

可选策略：

- 及时报错：转写或结构化失败时明确显示错误状态，但保留音频、原始文本和已识别字段，允许用户补全后保存。
- 静默降级：结构化失败时保留原始转写文本，不强打断用户，用户可继续补全字段。

用户后续可以在设置中修改失败处理策略。

### 3.8 日志状态

QSO 记录需要状态字段，状态变更由用户手动确认。

MVP 至少支持：

- 草稿：尚未满足正式保存条件，或用户尚未确认。
- 待确认：AI 已生成结构化结果，但仍需要用户复核。
- 已确认：用户确认字段后保存为正式 QSO。
- 已导出：记录已参与导出。
- 失败：转写或结构化失败，需要用户处理。

导出时用户可以按日志状态筛选。

## 4. AI 与语音供应商

### 4.1 基本原则

第一版在线优先。

当前阶段不要求内置离线语音模型或本地 LLM 推理能力，因为本地模型效果和设备适配不一定能满足实时记录要求。但架构和配置上应保留本地 endpoint 接入入口，便于未来接入局域网服务或本机服务。

### 4.2 接入方式

第一版采用“内置供应商模板 + 本地 endpoint 高级配置”的方式。

用户主要填写 API Key、模型名、base URL 等必要信息，不要求用户配置任意 HTTP Header、Body 模板或响应 JSONPath。

供应商配置分为两层：

- 供应商配置：配置供应商、API Key、base URL 等连接信息。
- 模型分配：分别为转录、结构化整理等步骤选择具体模型。

模型选择需要按能力联动：

- 仅支持文本的大模型不能出现在语音转录模型选择中。
- 不支持实时流式转写的模型不能出现在实时流式转写模型选择中。
- 不支持结构化整理的模型不能出现在结构化模型选择中。
- 用户添加某个供应商后，App 只在对应步骤展示该供应商可用且能力匹配的模型。

### 4.3 第一版供应商范围

第一版计划内置国内外常见供应商模板，包括但不限于：

- OpenAI-compatible
- 本地 endpoint
- OpenAI
- DeepSeek
- 通义千问 / Qwen
- 智谱
- Gemini

供应商边界：

- 真流式转写默认使用 OpenAI，但用户可以修改。
- 本地 endpoint 首版只考虑 OpenAI-compatible HTTP。
- OpenAI Realtime / WebSocket 可作为真流式转写的特例实现；OpenAI-compatible HTTP 用于结束后统一识别和本地 endpoint。
- 转写接口优先参考常见供应商文档，若 OpenAI 风格抽象可以覆盖首版需求，则优先采用 OpenAI 风格抽象。
- Gemini 保留。

## 5. 本地存储

除 HTTP 调用语音识别服务和大模型服务外，第一版数据默认存储在本地。

本地存储内容包括：

- QSO 记录
- 原始转写文本
- 原始音频文件
- 用户配置
- 供应商配置
- 导出历史或导出文件信息

原始数据保留策略：

- 默认保存原始音频和原始转写内容。
- 用户可以配置在完成识别后自动删除原始音频。
- 用户可以配置在日志确认后自动删除原始音频。
- 用户可以手动删除原始音频和原始转写内容。
- 本地数据不加密。

## 6. 导出

第一版必须支持：

- ADIF 导出
- CSV 导出
- 原始转写文本导出

字段映射：

- ADIF 字段映射规则在实现时参考 ADIF 规范和常见日志软件导出格式制定。
- CSV 字段列定义在实现时参考通用 QSO 日志导出格式制定。

导出筛选：

- 支持按日志状态筛选。
- 支持按日期、波段、模式等条件筛选。

## 7. 补录能力

主路径先做实时记录。

MVP 需要支持补录场景：

- 录音文件导入分析。
- 纯文本导入分析。

补录需要在 MVP 预留入口并实现基础流程。补录和实时记录的差异主要在输入来源，转写、结构化、确认、保存、导出流程应复用同一套处理链路。

## 8. 明确不进入 MVP 的内容

以下能力第一版不做，可作为后续 roadmap：

- 自动判断 QSO 开始/结束。
- 内置离线语音模型或本地 LLM 模型。
- 自动生成最终日志且不需要用户确认。
- 与 QRZ、LoTW、eQSL、HamQTH 等外部台站/日志服务同步。
- 比赛模式、POTA/SOTA 完整专项流程、奖状统计等高级业余无线电场景。
- POTA/SOTA Only 等专项筛选条件。
- ADIF 文件导入。
- 多设备云同步或账号系统。

## 10. 技术栈

- App：Flutter
- 数据存储：本地优先
- 网络调用：仅用于语音识别、LLM 结构化和用户配置的供应商 endpoint

### 10.1 本地数据库

第一版使用 SQLite 作为本地数据库。

### 10.2 平台节奏

首版先支持 Android。iOS 放到 MVP 后的小版本，鸿蒙兼容适配放到中长期方向。

### 10.3 语音实时采集与流式转写方案

语音处理链路采用分层、可替换的实时音频管线：

```text
Microphone Capture
 -> AudioFrame Stream(PCM16 mono)
 -> Speech Provider Adapter
 -> Transcript Segment Stream
 -> QSO Structuring Engine
 -> QSO Draft Store(SQLite)
 -> UI
```

基本原则：

- Flutter 侧负责采集标准音频帧，例如 PCM16、单声道、供应商指定采样率。
- 录音采集层与转写供应商解耦，业务层不直接依赖具体录音插件或具体云服务。
- 转写服务通过 Adapter 接入，OpenAI、Google、本地 endpoint、Gemini 等供应商都作为不同 Adapter。
- 结构化基于稳定转写片段增量更新 QSO 草稿，不对每个音频 chunk 都调用大模型。
- QSO 开始和结束仍由用户手动控制，VAD 只能用于音频切片或提交缓冲，不用于自动判断一条 QSO 的边界。
- 上层业务接口保持稳定，具体录音插件、转写供应商或结构化实现可以替换。

MVP 推荐实现：

- 音频采集优先采用 Flutter 生态中主流、活跃且支持 PCM stream 的录音能力，首选调研 record，若实际测试不满足再替换实现。
- 第一批优先实现 OpenAI Realtime / WebSocket 作为真正流式转写供应商。
- 本地 endpoint 作为高级配置接入，首版仅考虑 OpenAI-compatible HTTP。
- 对不支持真正流式的供应商，MVP 提供结束后统一识别模式作为通用 fallback。
- 分片准实时转写不进入 MVP，可放到 MVP 后小版本。
- 供应商配置中必须明确标识该供应商支持实时流式转写，还是只能结束后统一识别，不在 UI 上伪装成同一种能力。

建议抽象接口：

```text
AudioCaptureService
- start()
- frames: Stream<AudioFrame>
- stop()

SpeechTranscriptionProvider
- startSession(config)
- sendAudioFrame(frame)
- transcriptEvents: Stream<TranscriptEvent>
- commit()
- stop()

QsoStructuringService
- applyTranscript(segment, currentDraft)
- returns QsoDraftPatch
```

转写供应商需要声明能力：

```text
supportsStreaming
supportsPartialResult
supportsFinalSegment
preferredSampleRate
acceptedEncoding
requiresManualCommit
```

结构化处理采用 patch 更新模式，不直接让 AI 覆盖整条 QSO：

- 用户手动修改过的字段，AI 不能直接覆盖，只能提示。
- 低置信度字段标记为需确认。
- 原始转写片段需要保留，便于用户回看。
- 每次结构化 patch 记录来源片段，便于排查和复核。

### 10.4 状态管理

第一版状态管理采用 Riverpod。SQLite 负责持久化，Riverpod 负责 App 运行时状态、异步流程和服务依赖注入。

需要管理的主要状态包括：

- RecordingSessionState：空闲、请求权限、录音中、实时转写中、结束后识别中、结构化中、待确认、已保存、失败。
- TranscriptionModeState：实时流式转写、结束后统一识别。
- ProviderConfigState：供应商、API Key、base URL、可用模型列表。
- ModelAssignmentState：转录模型、结构化整理模型。
- CapabilityState：模型是否支持语音、流式转写、结构化整理。
- QsoDraftState：当前 QSO 字段、置信度、用户是否手动修改、字段来源转写片段。
- TranscriptState：原始转写片段、partial/final 状态、关联音频片段。
- LogStatusState：草稿、待确认、已确认、已导出、失败。
- ExportFilterState：按状态、日期、波段、模式筛选导出。

技术约束：

- UI 和业务流程依赖抽象接口，不直接绑定具体插件或供应商 SDK。
- 更换录音插件、转写供应商或结构化实现时，上层状态和业务接口不应变化。

## 11. 多语言与页面设计约束

### 11.1 多语言

首版支持简体中文和英文。

多语言规则：

- 默认语言跟随系统。
- 用户可以在设置中手动切换语言。
- 所有 UI 文案、状态名、按钮、错误提示、设置项、导出说明必须走本地化资源，不允许硬编码在页面组件中。
- QSO 字段标准名和无线电缩写可以保留英文缩写，例如 RST、QTH、ADIF、CSV、SSB、CW、FT8。
- ADIF/CSV 导出的字段名和格式不应因 UI 语言切换而破坏兼容性。
- Stitch 设计稿中的 HamLogger 仅作为视觉占位名称，不作为最终产品名；中文产品名以“通联随记”为准，英文展示名发布前再单独确认。

### 11.2 页面结构

首版页面结构参考 Stitch 生成稿，但需要按当前需求收敛：

- Record：实时记录主页面，优先优化操作效率。
- Logs：QSO 历史与筛选。
- Import：录音文件和纯文本补录。
- Export：ADIF、CSV、原始转写文本导出。
- Settings：转写模式、失败策略、音频保留、供应商配置、模型分配、语言设置、本地数据管理。

页面设计原则：

- 现代化、实用、操作便捷，不做花哨视觉。
- 信息密度服务于现场操作，不做营销式页面。

## 12. 验收口径草案

MVP 可用的最低验收口径：

- 用户能在移动端手动开始和结束一条 QSO。
- 用户能在简体中文和英文之间切换，默认语言跟随系统。
- 用户首次使用实时记录前能选择转写处理模式。
- 在实时流式转写模式下，App 能在通联过程中显示实时转写内容。
- 在实时流式转写模式下，App 能根据转写内容实时更新 QSO 草稿字段。
- 在结束后统一识别模式下，App 能先录音，并在用户结束 QSO 后提交转写和结构化。
- App 能在本次会话中展示已记录的 QSO 列表。
- App 能识别常见拼读字母形式的呼号。
- App 能对低置信度或不确定字段提示用户复核。
- 用户能在保存前检查并补全字段。
- 缺少呼号、日期时间、频率或波段、模式、发送和接收 RST 时不能保存正式 QSO。
- 用户能配置至少一种在线供应商完成语音转写和结构化处理。
- 用户能分别选择转录模型和结构化整理模型，且模型列表按能力过滤。
- 用户能配置本地 endpoint。
- 用户能导入录音文件或纯文本进行补录。
- 用户能查看并手动确认 QSO 日志状态。
- 用户能导出 ADIF、CSV 和原始转写文本。
- 用户能按状态、日期、波段、模式等条件筛选导出。
- QSO 记录和配置在本地保存。
